# ТЗ: Система коррекции медицинских терминов в VoiceMed

> Версия: 1.0
> Дата: 11.03.2026
> Проект: VoiceMed — голосовое заполнение медицинских документов

---

## 1. Описание проекта

**VoiceMed** — веб-приложение для врачей. Врач диктует голосом информацию о пациенте, система распознаёт речь (Whisper STT), извлекает структурированные данные (LLM) и заполняет медицинский документ.

**Текущий стек:**
- Фронтенд: React + TypeScript + Vite + Tailwind CSS
- Бэкенд: Node.js + Fastify + TypeScript
- STT (речь → текст): Whisper (HTTP-сервер на 192.168.41.161:9000)
- LLM (извлечение полей): Qwen3-8B через llama.cpp (HTTP-сервер на 192.168.41.161:8080)
- TTS (текст → речь): Python-сервер

---

## 2. Проблема

Whisper STT при транскрипции медицинской речи на русском языке допускает **систематические ошибки**, которые снижают качество итогового документа и требуют ручных правок от врача.

### 2.1. Типы ошибок

#### Ошибка 1: Латинские аббревиатуры превращаются в кириллицу

**Причина:** Whisper запущен с параметром `language: "ru"`, поэтому все звуки транскрибируются кириллицей. Латинские буквы подавляются.

| Врач говорит | Whisper выдаёт | Должно быть |
|---|---|---|
| "RA–ПТ" (радиочастотная абляция – пароксизмальная тахикардия) | РАПТ | RA–ПТ |
| "SpO2 девяносто восемь" | СПО2 98 | SpO2 98% |
| "NYHA два" | найха 2 | NYHA II |
| "CHA2DS2-VASc два балла" | чадс васк 2 балла | CHA₂DS₂-VASc 2 балла |
| "HAS-BLED один балл" | хас блед 1 балл | HAS-BLED 1 балл |

#### Ошибка 2: Лишние запятые

**Причина:** Whisper интерпретирует каждую паузу в речи как знак пунктуации. Врач задумался на 1-2 секунды — появляется запятая.

| Врач говорит (с паузами) | Whisper выдаёт | Должно быть |
|---|---|---|
| "АД... сто двадцать... на восемьдесят... ЧСС... семьдесят два" | АД, сто двадцать, на восемьдесят, ЧСС, семьдесят два, | АД 120/80 мм рт. ст., ЧСС 72 уд/мин |
| "Жалобы на... одышку... при физической нагрузке" | Жалобы на, одышку, при физической нагрузке, | Жалобы на одышку при физической нагрузке |

#### Ошибка 3: Арабские цифры вместо римских

**Причина:** Whisper всегда записывает числа арабскими цифрами. В медицине для функциональных классов, стадий и степеней принято использовать римские.

| Врач говорит | Whisper выдаёт | Должно быть |
|---|---|---|
| "ХСН функциональный класс два" | ХСН функциональный класс 2 | ХСН ФК II |
| "NYHA три" | NYHA 3 | NYHA III |
| "гипертоническая болезнь второй стадии" | ГБ 2 стадии | ГБ II стадии |
| "АВ-блокада первой степени" | АВ-блокада 1 степени | АВ-блокада I степени |
| "митральная недостаточность два" | МН 2 | МН II ст. |
| "CCS два" | CCS 2 | CCS II |

#### Ошибка 4: Орфографические искажения медицинских терминов

**Причина:** Whisper подбирает фонетически похожие слова из общей лексики, а не из медицинского словаря.

| Врач говорит | Whisper выдаёт | Должно быть |
|---|---|---|
| бисопролол | бесопролол | бисопролол |
| амлодипин | амлодепин | амлодипин |
| фибрилляция | фибриляция | фибрилляция |
| тахикардия | такая кардия | тахикардия |
| гипертензия | гипертензея | гипертензия |

#### Ошибка 5: Разрыв длинных медицинских терминов

**Причина:** Whisper вставляет дефисы или пробелы в длинные незнакомые слова.

| Врач говорит | Whisper выдаёт | Должно быть |
|---|---|---|
| ривароксабан | ривароксо-бан | ривароксабан |
| эмпаглифлозин | эмпа-глифлозин | эмпаглифлозин |
| спиронолактон | спироно-лактон | спиронолактон |
| тромбоэмболия | тромбо эмболия | тромбоэмболия |

### 2.2. Масштаб проблемы

- Ошибки встречаются в **каждой** диктовке
- Врач тратит **2-5 минут** на ручное исправление каждого документа
- Некоторые ошибки (латинские аббревиатуры) невозможно исправить настройками Whisper при `language: "ru"`

## 3. Архитектура решения

### 3.1. Общая схема пайплайна (до и после)

**СЕЙЧАС (текущий пайплайн):**
```
Врач диктует
    ↓
Whisper STT (распознаёт речь → сырой текст с ошибками)
    ↓
LLM (извлекает поля из текста, но работает с ошибочным входом)
    ↓
Документ в UI (врач видит ошибки, исправляет вручную, правки теряются)
```

**ПОСЛЕ (новый пайплайн):**
```
Врач диктует
    ↓
Whisper STT (распознаёт речь → сырой текст с ошибками)
    ↓
[НОВОЕ] Постобработка словарём замен
    │   • Базовый словарь: ~500 медицинских терминов (зашит в код)
    │   • Пользовательский словарь: замены, добавленные врачом через UI
    │   • Результат: исправлены латинские аббревиатуры, препараты, разрывы слов
    ↓
LLM (извлекает поля + исправляет пунктуацию, орфографию, римские цифры)
    │   • [НОВОЕ] Правила в промпте: убрать лишние запятые
    │   • [НОВОЕ] Правила в промпте: арабские → римские для ФК, стадий, степеней
    │   • [НОВОЕ] Правила в промпте: исправить орфографию мед. терминов
    ↓
Документ в UI
    ↓
[НОВОЕ] Врач видит ошибку → правый клик → попап → вводит правильный вариант
    │   • Текст заменяется в поле
    │   • Пара "ошибка → правильно" сохраняется в словарь
    │   • При следующей диктовке эта ошибка исправится автоматически
    ↓
Словарь растёт с каждым использованием системы
```

### 3.2. Три уровня коррекции

| Уровень | Что исправляет | Где работает | Скорость |
|---|---|---|---|
| **1. Словарь замен** | Латинские аббревиатуры, препараты, разрывы слов | Сервер, после Whisper | Мгновенно (~1мс) |
| **2. LLM-промпт** | Пунктуация, орфография, римские цифры | Сервер, при обработке LLM | Уже включено в запрос |
| **3. UI-коррекция врачом** | Всё остальное, что не поймали уровни 1 и 2 | Браузер | Ручной ввод врача |

Каждый уровень ловит то, что пропустил предыдущий. Уровень 3 обучает уровень 1 (замены сохраняются в словарь).

### 4.3. Алгоритм применения замен

```typescript
function applyMedicalDictionary(text: string): string {
  // 1. Загружаем все замены: базовые + пользовательские
  const allCorrections = [
    ...BROKEN_WORDS,           // Сначала склеиваем разорванные слова
    ...LATIN_ABBREVIATIONS,    // Потом латинские аббревиатуры
    ...CYRILLIC_CASE_FIXES,    // Потом регистр кириллических
    ...DRUG_NAME_FIXES,        // Потом препараты
    ...SPELLING_FIXES,         // Потом орфография
    ...userCorrections,        // В конце — пользовательские (высший приоритет)
  ];

  // 2. Применяем замены последовательно
  let result = text;
  for (const { wrong, correct } of allCorrections) {
    // Замена с учётом границ слов, регистронезависимая
    const regex = new RegExp(`\\b${escapeRegex(wrong)}\\b`, 'gi');
    result = result.replace(regex, correct);
  }

  return result;
}
```

**Важные детали:**
- `\b` (граница слова) — чтобы "экг" заменялось на "ЭКГ", но "рентгенографию" не ломало
- Порядок замен важен: сначала склеиваем разрывы ("ривароксо-бан" → "ривароксабан"), потом всё остальное
- Пользовательские замены идут последними и перезаписывают базовые

**Файл:** `server/src/services/whisper.ts`

```typescript
// БЫЛО:
const transcription = await whisperTranscribe(audioBuffer);

// СТАЛО:
const transcription = await whisperTranscribe(audioBuffer);
const corrected = applyMedicalDictionary(transcription);  // ← НОВОЕ
return corrected;
```

Функция вызывается **после** получения текста от Whisper и **до** отправки в LLM. Это занимает ~1мс и не влияет на производительность.

## 5. Компонент 2: Правила LLM-промпта

### 5.1. Что это

Дополнительные правила в системном промпте LLM, которые заставляют модель:
- Убирать лишние запятые
- Исправлять орфографию
- Конвертировать арабские цифры в римские в нужных контекстах

### 5.2. Изменения в файле: `server/src/services/llm.ts`

Добавить в `SYSTEM_PROMPT` следующие правила (после существующих):

```
Правила форматирования текста:
13. Пунктуация: убери лишние запятые, оставь только те, которые нужны по правилам русского языка.
    Whisper ставит запятую на каждой паузе — большинство из них лишние.
14. Орфография: исправь орфографические ошибки в медицинских терминах
    (например: "фибриляция" → "фибрилляция", "гипертензея" → "гипертензия").
15. Римские цифры — используй в следующих случаях:
    - Функциональные классы: ФК I, ФК II, ФК III, ФК IV
    - Классификация NYHA: NYHA I, NYHA II, NYHA III, NYHA IV
    - Классификация CCS: CCS I, CCS II, CCS III, CCS IV
    - Классификация EHRA: EHRA I, EHRA IIa, EHRA IIb, EHRA III, EHRA IV
    - Стадии ГБ: ГБ I стадии, ГБ II стадии, ГБ III стадии
    - Степени АВ-блокады: АВ-блокада I степени, II степени, III степени
    - Степени клапанной недостаточности: МН I ст., МН II ст., АН I ст. и т.д.
    НЕ используй римские для: СД 1 типа, СД 2 типа (тут арабские).
```

### 5.3. Почему LLM, а не словарь

Задачи пунктуации и римских цифр **нельзя решить простыми заменами**:
- Нужно понимать контекст: "ФК 2" → "ФК II", но "2 таблетки" → оставить "2 таблетки"
- Нужно понимать, какие запятые лишние, а какие нужны
- LLM уже обрабатывает текст — эти правила не добавляют дополнительного запроса

## 6. Компонент 3: UI — попап коррекции термина

### 6.1. Что это

React-компонент — небольшое всплывающее окно, которое появляется при правом клике на выделенном тексте в любом текстовом поле экрана редактирования. Позволяет врачу:
1. Увидеть ошибочный текст
2. Ввести правильный вариант

### 6.3. Поведение попапа (подробно)

**Открытие:**
1. Врач выделяет мышью слово или фразу в любом textarea на экране редактирования
2. Нажимает правую кнопку мыши
3. Вместо стандартного контекстного меню браузера появляется наш попап
4. Попап позиционируется абсолютно рядом с курсором мыши (снизу-справа от клика)
5. Если попап вылезает за край экрана — сдвигается так, чтобы быть полностью видимым
6. Поле "Распознано" заполняется выделенным текстом (только для чтения)
7. Курсор автоматически устанавливается в поле "Правильно"

**Работа:**
1. Врач набирает правильный вариант в поле "Правильно"
2. Галочка "Запомнить для будущих диктовок" — по умолчанию включена
3. Нажимает "Сохранить" (или Enter на клавиатуре)

**Закрытие:**
- По нажатию "Отмена"
- По нажатию Escape на клавиатуре
- По клику вне попапа (в любое место экрана)

**Что происходит при "Сохранить":**
1. Выделенный текст в textarea **заменяется** на введённый правильный вариант
2. Если "Запомнить" включено:
   - Отправляется POST-запрос на сервер: `{ wrong: "РАПТ", correct: "RA-ПТ" }`
   - Сервер добавляет эту замену в пользовательский словарь
   - При следующей диктовке "РАПТ" автоматически станет "RA-ПТ" ещё до LLM
3. Попап закрывается

### 6.4. Изменения в файле: `src/components/EditingScreen.tsx`

**Что добавляется:**

```typescript
// Новый стейт для попапа
const [correctionPopup, setCorrectionPopup] = useState<{
  visible: boolean;
  position: { x: number; y: number };
  selectedText: string;
  fieldKey: string;          // какое поле документа редактируется
  selectionStart: number;    // начало выделения в textarea
  selectionEnd: number;      // конец выделения в textarea
} | null>(null);

// Обработчик правого клика на textarea
function handleContextMenu(e: React.MouseEvent, fieldKey: string) {
  const textarea = e.target as HTMLTextAreaElement;
  const selected = textarea.value.substring(textarea.selectionStart, textarea.selectionEnd);

  if (selected.trim().length === 0) return; // Ничего не выделено — стандартное меню

  e.preventDefault(); // Отменяем стандартное контекстное меню браузера

  setCorrectionPopup({
    visible: true,
    position: { x: e.clientX, y: e.clientY },
    selectedText: selected.trim(),
    fieldKey,
    selectionStart: textarea.selectionStart,
    selectionEnd: textarea.selectionEnd,
  });
}

// Обработчик сохранения из попапа
async function handleCorrectionSave(wrong: string, correct: string, remember: boolean) {
  // 1. Заменяем текст в поле документа
  const field = correctionPopup.fieldKey;
  const currentText = document[field];
  const newText = currentText.substring(0, correctionPopup.selectionStart)
                + correct
                + currentText.substring(correctionPopup.selectionEnd);
  setDocument({ ...document, [field]: newText });

  // 2. Если "Запомнить" — отправляем на сервер
  if (remember) {
    await fetch('/api/corrections', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ wrong, correct }),
    });
  }

  // 3. Закрываем попап
  setCorrectionPopup(null);
}
```

**В JSX — у каждого textarea добавляется обработчик:**

```tsx
<textarea
  value={document.complaints}
  onChange={...}
  onContextMenu={(e) => handleContextMenu(e, 'complaints')}  // ← НОВОЕ
/>
```

## 7. Компонент 4: API-эндпоинты на сервере

### 7.1. Что это

Три HTTP-эндпоинта на Fastify-сервере для управления пользовательским словарём замен.

### 7.2. Изменения в файле: `server/src/server.ts`

#### Эндпоинт 1: Добавить замену

```
POST /api/corrections
Content-Type: application/json

Тело запроса:
{
  "wrong": "РАПТ",       // Что написал Whisper (ошибочный вариант)
  "correct": "RA-ПТ"     // Как должно быть (правильный вариант)
}

Ответ (200):
{
  "success": true,
  "id": "corr_1710150000_abc",   // уникальный ID замены
  "totalCorrections": 42          // общее количество замен в словаре
}

Ответ (400):
{
  "error": "Поля 'wrong' и 'correct' обязательны"
}
```

**Что происходит на сервере:**
1. Валидация: оба поля непустые, `wrong !== correct`
2. Проверка на дубликат: если замена для `wrong` уже существует — обновляем `correct`
3. Добавляем запись в массив в памяти
4. Записываем массив в файл `server/src/data/user-corrections.json`
5. Обновляем словарь в `medical-dictionary.ts` (без перезапуска сервера)

#### Эндпоинт 2: Получить все замены

```
GET /api/corrections

Ответ (200):
{
  "corrections": [
    {
      "id": "corr_1710150000_abc",
      "wrong": "РАПТ",
      "correct": "RA-ПТ",
      "createdAt": "2026-03-11T10:00:00Z"
    },
    {
      "id": "corr_1710150100_def",
      "wrong": "найха два",
      "correct": "NYHA II",
      "createdAt": "2026-03-11T10:01:40Z"
    }
  ],
  "total": 2
}
```

**Зачем:** Можно будет в будущем добавить страницу управления словарём, где врач видит все свои замены.

#### Эндпоинт 3: Удалить замену

```
DELETE /api/corrections/:id

Ответ (200):
{ "success": true }

Ответ (404):
{ "error": "Замена не найдена" }
```

**Зачем:** Если врач случайно добавил неправильную замену, он может её удалить.

### 7.3. Новый файл: `server/src/data/user-corrections.json`

Файл создаётся автоматически при первом добавлении замены. Формат:

```json
{
  "corrections": [
    {
      "id": "corr_1710150000_abc",
      "wrong": "РАПТ",
      "correct": "RA-ПТ",
      "createdAt": "2026-03-11T10:00:00Z"
    }
  ]
}
```

**Выбор JSON-файла вместо базы данных:**
- Простота: не нужно устанавливать и настраивать СУБД
- Достаточно для MVP: словарь врача — это ~50-200 замен, JSON справится
- Легко перенести: файл можно скопировать, забэкапить, отправить коллеге
- В будущем легко мигрировать на БД, если понадобится масштабирование

## 9. Хранение данных

### 9.1. MVP (текущая реализация — без базы данных)

| Что | Где хранится | Формат |
|---|---|---|
| Базовый словарь (~500 замен) | `server/src/services/medical-dictionary.ts` | TypeScript-массив, зашит в код |
| Пользовательские замены | `server/src/data/user-corrections.json` | JSON-файл на сервере |
| Кэш замен (оффлайн) | `localStorage` в браузере | JSON-строка |

**Как это работает при запуске сервера:**
1. Сервер стартует
2. Загружается базовый словарь из `medical-dictionary.ts` (~500 замен)
3. Загружается `user-corrections.json` (пользовательские замены)
4. Два словаря объединяются в один (пользовательские имеют приоритет)
5. При каждой транскрипции — объединённый словарь применяется к тексту

### 9.2. Будущее (с базой данных)

Когда/если приложение будет использоваться несколькими врачами:

```sql
CREATE TABLE corrections (
  id          TEXT PRIMARY KEY,
  wrong       TEXT NOT NULL,
  correct     TEXT NOT NULL,
  doctor_id   TEXT NOT NULL,       -- у каждого врача свой словарь
  created_at  TIMESTAMP DEFAULT NOW()
);
```

Миграция с JSON на БД будет простой — формат данных одинаковый.

```
Шаг 2 — Словарь замен:
(без изменений — здесь нет ошибок, которые ловит словарь)
```

## 13. Ограничения и риски

| Риск | Вероятность | Как минимизировать |
|---|---|---|
| Словарь заменит слово не в том контексте | Средняя | Использовать `\b` (границы слов) в regex, тестировать на реальных текстах |
| LLM не исправит все запятые | Низкая | Промпт с конкретными примерами, тестирование |
| Врачу неудобно выделять текст на мобильном | Высокая | На мобильном вместо правого клика — долгое нажатие (long press). Реализовать в следующей итерации |
| JSON-файл замен повредится | Низкая | Бэкап при каждой записи, валидация при чтении |
| Замена создаст новую ошибку (например, "МН" → что-то не то) | Средняя | Замены только по границам слов, возможность удалить замену через API |
